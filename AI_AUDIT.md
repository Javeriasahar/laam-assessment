# AI Audit Trail

Every AI-assisted action taken while building this project, in order. Tool: Claude (Anthropic), interactive chat with code execution (bash, file edit/create, build/test running).

| # | Action | AI Involvement | Manual Review / Changes |
|---|--------|----------------|--------------------------|
| 1 | Project scoping | Claude proposed stack + feature-scope options via clarifying questions before writing any code | Chose Next.js (single deployable app) and the fuller scope (core flow + alternatives + confidence score) |
| 2 | Project scaffold (`create-next-app`) | AI ran scaffolding command with TypeScript/Tailwind/App Router | Verified config output |
| 3 | DB library choice | AI attempted `better-sqlite3`, hit a native-compile failure (sandboxed network blocked nodejs.org header download) | AI diagnosed and switched to in-memory TS data instead of fighting the sandbox — a real, logged tradeoff rather than silently retrying |
| 4 | Data model design (`lib/types.ts`) | AI proposed the `Product`/`Variant`/`PricePoint`/`DeliveryZone`/`ConfidenceResult` shape | Reviewed for completeness against the brief's 5 customer questions (availability, size, price, delivery trust, alternatives) |
| 5 | Seed data (`data/seed.ts`) | AI wrote 6 initial products with deliberately varied stock/price/delivery scenarios so the scoring logic would have real signal to react to | Later added a 7th product (see #10) after testing surfaced a gap |
| 6 | Confidence scoring engine (`lib/confidence.ts`) | AI designed and implemented the weighted scoring model (stock 35 / delivery 30 / price 20 / safety-net 15) with documented rationale in code comments | Reviewed the weighting logic for reasonableness; this is the one invented business rule in the app |
| 7 | API routes | AI implemented `/api/products`, `/api/products/:id`, `/api/products/:id/confidence` | Verified query param validation (size/zone allowlists) |
| 8 | Frontend components + pages | AI built `ConfidenceDial`, `ConfidenceFactors`, `SizeSelector`, `DeliverySelector`, `Alternatives`, product page, home page | Reviewed against the required user flow (discover → check size/price/delivery → decide) |
| 9 | Visual design | AI followed an internal design-system process (palette/type/layout plan reviewed against "does this look like a generic AI template" before building) to land on an editorial green/rose/bronze identity instead of default AI-template styling | Accepted; matches a fashion-marketplace subject better than a generic dashboard look |
| 10 | Build + runtime verification | AI ran `npm run build`, started the dev server, and used `curl` to exercise the live API with real query combinations rather than assuming the code worked | **This step caught a real bug**: `curl .../p3/confidence?size=M&zone=major_city` returned "medium" confidence (65/100) for a product where the selected size had zero stock — the weighted formula let strong delivery/price scores mask a hard blocker. Instructed AI to add an override (`out-of-stock -> level = "low"`) and a regression test. Also caught: alternatives list was empty for that same product because the seed data only had one item in its category — instructed AI to add a second matching product |
| 11 | Test suite | AI wrote 12 unit tests for `lib/confidence.ts` covering the scoring engine and alternatives matching, including a dedicated test for the bug found in #10 | Ran `npx vitest run` myself and confirmed all 12 pass; would have failed pre-fix, confirming the test is a real regression guard and not padding |
| 12 | README | AI drafted all 9 required sections | Reviewed for accuracy against what was actually built and tested, not aspirational claims |

| 13 | Added explicit "Why?" breakdown per user request | AI rewrote factor detail strings to be single self-contained lines (rather than label+detail pairs), added a "Why?" heading to the factors component, added a new "alternate sizes available" factor when the chosen size is out but others aren't, and added a regression test for it | Verified live API output against the two example formats requested, confirmed near-exact match; re-ran full test suite (13/13 pass) and build |

| 14 | Redesigned placeholder images per user request for a more polished look | First attempt was a hand-drawn garment-silhouette SVG path; AI rendered it to PNG to check the result and found the path didn't read as a coherent garment shape. Rejected that approach and rebuilt as framed "fabric look card" swatches instead — a pattern style (floral/stripe/lattice/geometric) matched to each product's actual fabric/print tags, with brand mark and fabric label, plus a second close-up crop image | Rendered every image to PNG via cairosvg and visually inspected each one before accepting; this is a second concrete example of catching and correcting a bad AI-generated output rather than shipping it unreviewed |

**Overall verification approach:** rather than accepting generated code as final, every stage that could fail silently (build, runtime, API responses, and now visual output) was actually executed and inspected before moving on. Two real issues were caught this way: the out-of-stock scoring bug (#10) and the broken garment-silhouette illustration (#14) — neither was found by reading code alone.
